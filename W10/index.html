§0 · Why do we need CNNs?

Before we touch a single problem, let's understand why convolutional networks exist at all. The notes give us a striking example: a single MLP layer that maps a 224×224×3 image to 1024 features needs over 150 million parameters. Let's see this live.

A single MLP layer on a moderately-sized image already eats hundreds of megabytes of memory just for parameters. Doubling the image side quadruples this. This is unsustainable, and it ignores all the other layers we'd want to stack on top.
CNNs solve this with two big ideas:

1. Local connectivity — a neuron only looks at a small spatial region of the input, not the entire image.

2. Weight sharing — the same small filter is slid across the whole image, so we learn one set of weights that detects a feature anywhere it appears.

These two inductive biases reduce parameter counts by factors of thousands.

§1 · Background: what is convolution doing?

Before we compute, let's make sure we agree on the picture. A convolution slides a small kernel across the input, and at each position computes a weighted sum.

The output-size formula — derive, don't memorize

Hout = ⌊ (Hin + 2P − K) / S ⌋ + 1

Let's understand why this formula looks the way it does. Consider an input of size Hin and a kernel of size K, with no padding and stride 1.

1The kernel's leftmost position is index 0. Its rightmost valid position is Hin − K (any further right and the kernel falls off the edge).

2That gives us a sliding range of Hin − K. We step through it with stride S, so we get (Hin − K) / S steps.

3Add 1 for the starting position. Padding P added on each side effectively grows the input to Hin + 2P, hence the 2P in the numerator.

Feature Map A contains the feature. Its responses are roughly 4× larger in magnitude than Map B's. The kernel matches strongly somewhere in A.
"The output values are mostly negative — how can we say there's a feature?"

A negative response means the kernel matched but with inverted polarity. The kernel detects bright-top-left + dark-bottom-right patterns. A strong negative response means the opposite: dark-top-left + bright-bottom-right (an anti-pattern). Either way, the magnitude tells us a strong structured pattern exists. After a ReLU activation, negative responses would be clipped — but the unactivated magnitude is what the convolution layer alone produces.

Padding is not free. The zeros you insert at the border interact with real pixel values and can create phantom features the network sees but that don't exist in the data. In practice this matters less for trained networks (they learn to discount edges), but it's a real cost — and it's why alternatives like reflection padding or replication padding are sometimes used instead.
"Then why is zero-padding still the default in most CNN frameworks?"

Three reasons: (1) it's computationally simple — just allocate zeros, (2) it preserves the spatial dimensions cleanly so layer designs are predictable, and (3) the network learns to compensate during training. The trade-off is acceptable for most tasks, but for high-precision tasks like segmentation at boundaries, people switch to other padding modes.
Padding's main purpose is to control output size (often to keep spatial dimensions equal to input, called a "same convolution"). But it always introduces some boundary artifact. The trade-off — losing spatial information at edges vs. introducing artifacts — is a design choice every architect makes.

Think of it this way: a filter doesn't process channels separately and output separate channel responses. Instead, each filter is a single "feature detector" whose definition spans all input channels. To detect a "red-edge," you'd build a filter where Channel R has edge-detector weights and Channels G and B have small or zero weights. The summation across channels integrates all the color information into one decision.
"So how do we get multiple output channels?"

By using multiple filters. If you want 64 output channels, you build 64 separate filters (each of size 2×2×2 in our example). Each filter produces its own 2×2 output map, and you stack these into a 2×2×64 output tensor. This is exactly what the AlexNet parameter table is counting: K × K × Cin × Cout — one set of K×K×Cin weights per output channel.

§5 · Problem 4 — Counting FLOPs

This problem shows numerically why AlexNet's first layer uses stride 4. Let's derive the FLOP formula step by step, then play with the parameters.

Step 1: FLOPs per output pixel

To compute one element of the output:

1Multiplications: The kernel has C·k·k weights. Each is multiplied with one input value. → C·k² multiplications

2Additions: Adding n numbers takes n−1 additions (think: a+b is one operation, then +c, then +d, ...). → C·k² − 1 additions

3Total per pixel: 2C·k² − 1 FLOPs

Step 2: Multiply by output pixel count

Total FLOPs (one filter) = (2Ck² − 1) × Hout × Wout
Total FLOPs (full layer) = nfilters × (2Ck² − 1) × Hout × Wout

Step 3: Part (b) — AlexNet conv1 with stride 1, no padding

Given: C = 3, H = W = 224, k = 11, n = 64, S = 1, P = 0

Output size: (224 − 11 + 0)/1 + 1 = 214

FLOPs per filter:

(2 · 3 · 11² − 1) × 214²
= (726 − 1) × 45,796
≈ 2 · 3 · 121 · 214²   (approximately, ignoring the −1)
= 33,202,100 FLOPs ≈ 33M

For 64 filters: 64 × 33M = 2,124,934,400 ≈ 2.1G FLOPs

Step 4: Part (c) — AlexNet conv1 with stride 4, padding 2

Output size: ⌊(224 − 11 + 4)/4⌋ + 1 = ⌊217/4⌋ + 1 = 54 + 1 = 55

FLOPs per filter:

(2 · 3 · 11² − 1) × 55²
= 725 × 3,025
= 2,193,125 ≈ 2.2M FLOPs

For 64 filters: 64 × 2.2M = 140,360,000 ≈ 140M FLOPs

By changing stride from 1 to 4, we dropped from 2.1G to 140M FLOPs — roughly a 15× reduction. This is why AlexNet's first layer uses aggressive striding: the early layers operate on full-resolution input, where computation is most expensive.

"If padding adds pixels and stride removes them, don't they just cancel out?"

No, they serve different purposes:
Padding recovers information at the borders so edge pixels are seen by the filter multiple times. Without padding, the corner pixel is touched only once; with padding, it appears in several filter positions.
Stride deliberately skips positions to downsample. It throws away spatial information to save computation.
In AlexNet's conv1, padding 2 brings the effective input to 228×228, then stride 4 samples 55 positions across — using more of the border information while still aggressively downsampling.

This is the most challenging problem, but the core message is simple: backprop is just the chain rule applied carefully. We'll walk through the 1D case so the structure is crystal clear.

Step 2: Part (a) — Forward pass with stride 1, padding 0

Output length = (6 − 2)/1 + 1 = 5

z[0] = Y·A + Z·B
z[1] = Y·B + Z·C
z[2] = Y·C + Z·D
z[3] = Y·D + Z·E
z[4] = Y·E + Z·F

Step 3: Part (b) — Forward pass with stride 2, padding 0

Output length = (6 − 2)/2 + 1 = 3

z[0] = Y·A + Z·B    (kernel at position 0)
z[1] = Y·C + Z·D    (kernel at position 2)
z[2] = Y·E + Z·F    (kernel at position 4)

Notice the kernel only touches inputs in non-overlapping pairs because stride = kernel size. Some inputs (B, D, F) are touched once, just like A, C, E.

Step 4: Part (c) — Compute ∂L/∂Y using the chain rule

Y is a single weight, but it appears in every output element z[0], z[1], z[2]. Changing Y changes all three outputs simultaneously, and each output then affects the loss. The chain rule must sum over every path from Y to L.

The path diagram

Y ──→ z[0] ──→ L
 ├──→ z[1] ──→ L
 └──→ z[2] ──→ L

Y reaches the loss through three intermediate variables. We sum their contributions.

∂L/∂Y = ∂L/∂z[0] · ∂z[0]/∂Y  +  ∂L/∂z[1] · ∂z[1]/∂Y  +  ∂L/∂z[2] · ∂z[2]/∂Y

Compute each ∂z[i]/∂Y

OutputFormula∂z[i]/∂YWhy
z[0]Y·A + Z·BAY is multiplied with A; the partial isolates that term
z[1]Y·C + Z·DCY is multiplied with C
z[2]Y·E + Z·FEY is multiplied with E

Plug in

∂L/∂Y = A·∂L/∂z[0] + C·∂L/∂z[1] + E·∂L/∂z[2]

The upstream gradients (∂L/∂z[i]) come from the next layer up during backprop — they were computed before we got here. Our job in this layer is to combine them with the local partials.

Step 5: Part (d) — Compute ∂L/∂B

Now we go after the input gradient. We need to know: which output(s) does B appear in?

Looking at the stride-2 outputs:

z[0] = Y·A + Z·B    ← B appears here
z[1] = Y·C + Z·D    (no B)
z[2] = Y·E + Z·F    (no B)

So B has only one path to the loss, via z[0]. The chain rule has a single term:

∂L/∂B = ∂L/∂z[0] · ∂z[0]/∂B = Z · ∂L/∂z[0]
"What if stride were 1? Would B have appeared in more outputs?"

Yes! With stride 1 and kernel length 2, B appears in two outputs: z[0] = Y·A + Z·B and z[1] = Y·B + Z·C. Then ∂L/∂B = Z·∂L/∂z[0] + Y·∂L/∂z[1]. The stride directly controls how many outputs each input contributes to, which controls how many gradient terms get summed in backprop.

Step 6: Two beautiful patterns to leave students with

Pattern 1: weight gradients = convolution of input with upstream gradient.

Look at ∂L/∂Y = A·∂L/∂z[0] + C·∂L/∂z[1] + E·∂L/∂z[2]. The inputs A, C, E (in stride-2 positions) are multiplied with the corresponding upstream gradients and summed. This is itself a convolution-like operation. Hence the saying: "backprop through a conv layer is another conv."
Pattern 2: weights are shared, so their gradients accumulate.

Because Y is used at every output position, its gradient is a sum of many terms — one per output. This parameter sharing is the heart of why CNNs can learn so efficiently: a single weight gets updated based on evidence from many spatial locations.

Parameter count verification

The parameter formula from the notes is:

Parameters per layer = (KH × KW × Cin) × Cout + Cout

Let's verify Conv1: (11 × 11 × 3) × 96 + 96 = 121 × 3 × 96 + 96 = 34,848 + 96 = 34,944. Matches the notes exactly. Compare to the 154 million for a single MLP layer at the start of this lecture — Conv1 is over 4,000× smaller.

The big picture

"Why is Conv1 alone allowed to use a huge 11×11 kernel, but every later layer uses 3×3?"

Two reasons, both about efficiency:

Receptive field at low cost. Stacking two 3×3 convolutions gives the same receptive field as one 5×5 (each pixel in the second layer sees a 5×5 neighborhood of the input). Three stacked 3×3 layers match a 7×7 receptive field. But the parameter count is much smaller: two 3×3 layers have 2·9 = 18 weights per channel, while one 5×5 layer has 25.

More nonlinearities. Each conv layer is followed by ReLU, so stacking 3×3 layers gives the network more places to apply nonlinearity, increasing its representational power.

Conv1 is an exception because input images have only 3 channels, so 11×11×3 is still cheap, and the goal there is to capture coarse low-level structure (edges, gradients) at large spatial scale.
